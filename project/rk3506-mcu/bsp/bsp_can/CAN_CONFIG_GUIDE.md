# RK3506 CAN 外总线联调复盘（最终版）

## 1. 最终结果

本轮联调最终达到的状态是：

- MCU 周期发送 `ID=0x200` 的标准数据帧，CAN 分析仪可以稳定接收。
- CAN 分析仪发送 `ID=0x201` 的标准数据帧，MCU 侧已经可以成功接收。
- 当前稳定版本的串口标识为：
  `CAN Build Tag: 2026-04-09-CAN-EXT-1M-HARDIRQ-NOSELFTEST-NOFILTER`

这份文档记录的是最终有效结论，不再保留中间阶段那些“能临时发通但方向不对”的方案。

## 2. 问题演进

这次问题不是单点故障，而是几个问题前后叠加。

### 阶段 A：最开始中断链路不稳定

早期现象是：

- `Adp=0 / Can=0 / RxDisp=0`
- 看起来像 CAN 中断没有真正进入公共处理流程
- 一度还出现过异常时钟读数

这一阶段主要问题是中断链路和时钟基线都不稳定，导致收发现象混杂在一起，不容易判断。

### 阶段 B：MCU 能发，分析仪能收，但分析仪发给 MCU 失败

后面切到外总线后，出现了一个更有价值的现象：

- MCU 发 `0x200`，分析仪能收到
- 分析仪发 `0x201`，分析仪侧显示“发送失败”
- MCU 的 `REC` 会涨，但 `TEC` 基本不涨

这说明：

- MCU 的发送路径已经基本正常
- 真正出问题的是 MCU 的接收路径
- 不能再把“发送正常”误当成“整条 CAN 配置都正常”

### 阶段 C：去掉误导项后，定位到 RX 侧敏感问题

在删掉 `SELF_TEST`、删除 `CAN_Service_Poll()` 轮询兜底、切回纯硬件中断后，问题继续收敛：

- MCU 自发帧仍然正常
- 分析仪发帧给 MCU 仍失败
- `ERROR_CODE` 曾出现过 `RX/FORM`
- `RXST=0x4000`，对应接收状态机在 `ACK/ACK delimiter` 附近出错

这一步非常关键，因为它把问题从“软件逻辑”缩小到“接收侧采样/滤波/同步”。

## 3. 最终根因

### 根因 1：24MHz 回退时钟下，不能直接照搬 HAL 默认位时序

HAL 内置 `CANFD_BPS_1MBAUD` 的参数默认是按常规高速时钟档位设计的。  
当实际 `SCLK` 回退到 `24MHz` 时，如果不手工改位时序，1Mbps 不一定真的成立。

最终有效做法：

- 检测到 `SCLK` 在 `24MHz` 档时
- 手工覆写 nominal/data timing
- 使用 `brq=0, tseg1=8, tseg2=1`

这样总线实际速率才真正落在 1Mbps。

### 根因 2：`SELF_TEST` 不适合作为真实外总线的默认配置

曾经的中间方案里，为了在“单节点”场景下也能把帧发出去，给外总线模式保留了 `SELF_TEST`。  
这会带来一个很大的副作用：

- MCU 在接收外部节点帧时，不再按真实总线节点那样参与 ACK 行为
- 对端可能在 ACK 槽看不到期望响应
- 分析仪发给 MCU 时就会出现“发送失败”或重发

结论是：

- `SELF_TEST` 只能用于非常特定的自测/单节点场景
- 真实外总线联调默认必须关闭

### 根因 3：`CAN_Service_Poll()` 轮询兜底会干扰真实问题判断

轮询兜底在“先把链路打通”阶段确实有价值，但它不适合留在最终版里：

- 会把“是否真靠硬中断工作”这个问题掩盖掉
- 可能和中断里的 `INT` 清除产生竞争
- 发送完成/失败到底是硬件事件还是软件补救，不再清晰

最终版已经彻底去掉 `CAN_Service_Poll()`，改成纯中断驱动。

### 根因 4：RX 波形滤波和 ACK 槽重同步会伤到接收方向

这是本次联调里最容易忽视、但最后真正解决问题的一点。

在问题仍然存在时，寄存器里能看到：

- MCU 发给分析仪正常
- 分析仪发给 MCU 失败
- `REC` 递增
- `WAVE_FILTER_CFG = 0x308`

这说明：

- 发送侧没问题
- 接收侧存在额外处理
- 问题更像接收链路对 1Mbps 信号过于敏感

最终有效修正有两项：

- 关闭 `WAVE_FILTER_CFG`
- 在 24MHz 档位下置位 `ACKSLOT_SYNC_DIS`

前者去掉了 RX 波形滤波，后者避免 ACK 段的硬同步重新对齐。  
这两项都只作用于接收方向，和最终现象完全一致。

## 4. 最终稳定方案

### 4.1 总线模式

- 外总线模式默认关闭 `SELF_TEST`
- loopback 时只打开 `RXSTX`
- 不再保留“外总线也默认开 self-test”的逻辑

对应代码：

- `CAN_ApplyModeFlags()`

### 4.2 位时序

- 先尝试把 CAN 时钟设为 `200MHz`
- 如果读回异常，就回退到 `24MHz`
- 在 `24MHz` 档位手工覆写 1Mbps nominal/data timing
- 同时关闭 ACK 槽硬同步

对应代码：

- `CAN_InitController()`
- `CAN_ApplyNominalTiming1M()`

### 4.3 接收侧设置

- `WAVE_FILTER_CFG = 0`
- `RXINT_CTRL = 1`
- `ATF_CTL = ATF0_EN`

说明：

- `ATF_CTL` 只决定硬件接收过滤器是否打开
- 应用层是否真正把帧分发给模块，还要看 `CAN_Register()` 里配置的 `rx_id`

### 4.4 中断模型

最终版已经改为纯硬件中断：

- 放行 RX 相关中断
- 放行 TX 完成/失败相关中断
- `CAN_Common_IRQHandler()` 在 ISR 里更新 TX 完成/失败状态
- `CAN_Transmit()` 只等待 ISR 更新的状态位
- 不再使用 `CAN_Service_Poll()`

这样发送状态与硬件事件一一对应，排查时更可信。

## 5. 这次最有价值的诊断信号

### 5.1 `REC` 和 `TEC` 的意义完全不同

- `REC` 是接收错误计数，不是“收到多少帧”
- `TEC` 是发送错误计数

如果看到：

- `REC` 增长
- `TEC` 基本不变
- MCU 自己发出去的帧还能被分析仪收到

优先怀疑 RX 侧，不要先怀疑 TX。

### 5.2 `ERROR_CODE` 要结合方向看

不能只看 `ERROR_TYPE`，一定要同时看 `ERROR_DIRECTION`：

- `TX/...` 才是真正的发送方向错误
- `RX/...` 说明错误发生在接收方向

本次有代表性的现场是：

- `RX/FORM`
- `RXST=0x4000`

这把问题明确指向了接收状态机，而不是发送路径。

### 5.3 “分析仪显示发送失败”并不一定是分析仪的问题

当分析仪发给 MCU 失败时，很多人第一反应会去看分析仪配置。  
这次的经验是：

- 如果同时看到 MCU 发给分析仪是成功的
- 而分析仪发给 MCU 失败

那更该先看 MCU 的 RX 路径设置，而不是一味怀疑分析仪。

## 6. 最终代码策略

当前代码策略可以概括为：

- 发送方向：纯硬件中断确认
- 接收方向：去掉 RX 波形滤波，尽量保留原始总线波形
- 时序方向：24MHz 档位手工保证 1Mbps
- 模式方向：真实外总线不使用 `SELF_TEST`

这个组合是当前实测有效的稳定配置。

## 7. 这次踩坑后的经验总结

### 经验 1：先区分“谁发谁收”再排查

不要把 CAN 现象混成一句“收发不通”。  
至少要先区分：

- MCU 发 -> 分析仪收 是否正常
- 分析仪发 -> MCU 收 是否正常

这次真正的突破点，就是发现“前者正常、后者异常”。

### 经验 2：不要把临时调通方案直接当最终方案

`SELF_TEST` 和轮询兜底都曾经帮助快速缩小问题范围，  
但它们并不适合直接留在最终外总线版本里。

临时方案的目标是“帮助判断方向”，不是“直接作为量产配置”。

### 经验 3：接收侧专有配置值得单独怀疑

当出现“我能发出去，但我收不到别人”的情况时，要优先看：

- RX 波形滤波
- 接收重同步
- 接收采样点
- 接收过滤器

因为这些设置本来就只影响接收方向。

### 经验 4：日志里一定要保留硬件寄存器证据

这次能比较快地收敛问题，靠的是日志里一直保留了：

- `REC/TEC`
- `ERROR_CODE`
- `STATE`
- `MASK`
- `SCLK`

没有这些寄存器证据，很多结论都会停留在猜测。

### 经验 5：应用层过滤和硬件接收是两回事

CAN 分析仪是否显示“发送成功”，看的是总线 ACK。  
而 `RxDisp` 是否增长，看的是：

- 帧有没有进 MCU 接收存储
- `rx_id` 是否命中应用层分发条件

两者不是同一个问题，排查时不要混淆。

## 8. 后续建议

### 建议 1：保留当前稳定配置作为基线

如果后续还要继续调优位时序或滤波器，建议先以当前版本为基线，不要回到旧的 `SELF_TEST + Poll` 方案。

### 建议 2：后续如果要做更强鲁棒性测试

可以继续做这些验证：

- 不同分析仪/不同收发器交叉联调
- 长线/短线场景测试
- 双端 120 欧终端确认
- 24MHz 和其他时钟档位分别验证

### 建议 3：文档和代码一起维护

这次复盘里已经证明，旧文档非常容易因为中间状态遗留而误导后续排查。  
以后只要结论变了，就要同步更新文档。

## 9. 一句话结论

这次问题最终不是“CAN 发送不通”，而是：

**在 24MHz 时钟回退条件下，MCU 的 RX 路径对外部来帧过于敏感；去掉 `SELF_TEST`、改为纯硬件中断、关闭 RX 波形滤波、关闭 ACK 槽硬同步后，外总线双向通信恢复正常。**
