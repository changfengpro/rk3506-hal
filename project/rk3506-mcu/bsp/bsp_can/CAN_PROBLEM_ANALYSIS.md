# RK3506 CAN 问题复盘

## 1. 问题背景

本次问题发生在 RK3506 MCU 侧 `CAN0` 外总线联调过程中。

联调目标是：

- MCU 周期发送标准帧，`TX ID = 0x200`
- CAN 分析仪向 MCU 发送标准帧，`RX ID = 0x201`
- MCU 侧通过纯硬件中断完成收发状态更新
- RPMsg 维持原有正常状态，不因为 CAN 初始化被破坏

最终现象已经恢复正常：

- MCU -> 分析仪：可以稳定发送
- 分析仪 -> MCU：可以稳定进入回调
- `rx_buff` 和 `rx_len` 可以被正确更新
- `can tx err` 不再作为“明明发出去了却报错”的假故障长期出现

---

## 2. 当时的异常现象

本轮最后阶段的异常现象有两个，分别来自发送侧和接收侧。

### 2.1 发送侧现象

- 串口持续打印 `can tx err`
- 但 CAN 分析仪实际上可以收到 MCU 发出的报文

这说明：

- 物理总线发送并没有完全失败
- 更像是软件侧“发送完成判定”出了问题

### 2.2 接收侧现象

- CAN 分析仪向 MCU 发送报文时，分析仪显示发送成功
- CAN 控制器寄存器里能看到接收相关痕迹
- 但 `rx_buff` 一直是全 0
- `rx_len` 一直是 0
- 应用层回调没有看到有效接收结果

这说明：

- 硬件并不是完全没收到东西
- 问题更像发生在“驱动分发到业务实例”或者“中断状态更新”这一层

---

## 3. 直接证据

当时调试信息里有几个非常关键的证据。

### 3.1 说明硬件收到过数据

寄存器现场中可以看到：

- `INT = 0x1`
- `FD_RXID = 0x200`
- `RX_FIFO_RDATA` 非 0

这些现象说明 CAN 控制器内部已经出现过接收事件，不是单纯的物理层完全不通。

### 3.2 说明应用层没有拿到数据

同时又能看到：

- `rx_buff` 全 0
- `rx_len = 0`

这说明问题不在“总线上有没有波形”，而在“收到的帧有没有被正确分发到业务实例”。

### 3.3 说明发送错误是软件判定错误

用户现场表现是：

- 分析仪确实能收到 MCU 发出的帧
- MCU 串口却持续打印 `can tx err`

这类现象通常意味着：

- 发送动作已经发生
- 但软件等待的“发送完成/失败状态位”没有按预期更新

---

## 4. 最终根因

这次最终不是单一根因，而是两个问题叠加。

### 根因 1：应用层 `RX ID` 配错

最后阶段 `main.c` 里的测试配置一度变成了：

- `APP_CAN_TX_ID = 0x200`
- `APP_CAN_RX_ID = 0x200`

但实际联调约定是：

- MCU 发送给分析仪：`0x200`
- 分析仪发送给 MCU：`0x201`

也就是说，业务层希望接收的是 `0x201`，但代码里注册成了 `0x200`。

而当前 `bsp_can` 的接收分发逻辑是严格按下面两个条件同时匹配的：

1. `can_handle` 相同
2. `rx_id == rx_msg->stdId`

只有都匹配时，驱动才会：

- 拷贝到 `instance->rx_buff`
- 更新 `instance->rx_len`
- 调用 `can_module_callback`

因此当分析仪发 `0x201` 给 MCU 时，即使硬件已经把帧收进 FIFO，只要业务实例注册的 `rx_id` 不是 `0x201`，应用层依然会看到：

- `rx_buff` 不更新
- `rx_len` 维持 0
- 回调不生效

这就是“硬件像是收到了，但业务层像完全没收到”的直接原因。

### 根因 2：默认中断策略只开了 RX，没有开 TX

驱动里之前的默认配置是：

```c
static uint32_t s_canInterruptEnableMask = CAN_INTERRUPT_RX;
```

这意味着默认只放行 RX 中断，不放行 TX 完成/失败相关中断。

但是当前发送接口 `CAN_Transmit()` 的完成判定逻辑依赖下面两个状态变量：

- `s_canTxDone[devIdx]`
- `s_canTxFail[devIdx]`

而这两个状态变量是在 `CAN_Common_IRQHandler()` 里通过 TX 相关中断更新的。

所以当 TX 中断没有放行时，就会出现一个非常典型的现象：

1. `HAL_CANFD_Transmit()` 已经把报文写进控制器
2. 总线上确实发出去了，分析仪也确实收到了
3. 但 MCU 软件侧一直等不到 `TX_FINISH` 或 `TX_FAIL` 的状态更新
4. `CAN_Transmit()` 最后超时返回 0
5. 上层打印出 `can tx err`

也就是说，之前的 `can tx err` 很大一部分并不是“总线真的没发出去”，而是“软件等待发送结果的中断条件不成立”。

---

## 5. 为什么这两个问题叠加后看起来特别像硬件故障

这次最容易误导排查方向的地方在于：

- 一边是“分析仪能收到 MCU 的帧”，说明总线发送正常
- 另一边是“MCU 仍然报 can tx err”，看起来又像发送失败
- 同时“分析仪发给 MCU 成功”，但 `rx_buff` 又一直没数据

这种组合特别容易让人误以为：

- 中断链路全坏了
- CAN 控制器状态机异常
- 或者总线时序还有隐藏问题

实际上最后这轮问题里，更核心的是：

- 发送错误日志是假故障，根因是 TX 中断没放行
- 接收不到业务数据是假象，根因是业务实例 `rx_id` 配错

也就是说，这一轮最后卡住的并不是物理层，而是“驱动判定逻辑 + 业务层过滤条件”。

---

## 6. 最终修复

### 6.1 修复接收 ID

在 `main.c` 中把测试配置恢复为：

```c
#define APP_CAN_TX_ID  0x200U
#define APP_CAN_RX_ID  0x201U
```

这样业务实例才会正确接收来自分析仪的 `0x201` 报文。

### 6.2 显式打开 TX + RX 中断

在 `CAN_Service_Init()` 之前调用：

```c
CAN_SetInterruptEnable(CAN_INTERRUPT_ALL);
```

这样发送完成/失败状态就能通过 ISR 正常更新。

### 6.3 调整驱动默认值

为了避免以后再次出现“默认只开 RX，导致发送明明成功却报错”的问题，驱动默认值也改成了：

```c
static uint32_t s_canInterruptEnableMask = CAN_INTERRUPT_ALL;
```

这样即使应用层忘记显式配置，中断策略也更符合当前测试和常见业务预期。

---

## 7. 这次问题的本质结论

这次问题最终可以归纳为两句话：

### 结论 1

`rx_buff` 没数据，不一定代表 CAN 控制器没收到帧。

还可能是：

- 业务实例注册的 `rx_id` 不匹配
- 驱动已经收到，但没有把该帧分发给当前实例

### 结论 2

`can tx err` 不一定代表报文没有上总线。

还可能是：

- 总线已经发出去了
- 对端也收到了
- 只是软件侧没有收到 TX 完成/失败中断，导致等待超时

---

## 8. 后续联调建议

以后如果再遇到“CAN 看起来不通”，建议按下面顺序判断。

### 8.1 先分开看发送和接收

不要笼统地说“CAN 不通”，要先明确：

- MCU -> 分析仪 是否正常
- 分析仪 -> MCU 是否正常

### 8.2 再分开看硬件层和业务层

如果寄存器里已经看到：

- `INT` 有事件
- `FD_RXID` 有值
- `RX_FIFO_RDATA` 非 0

但应用层还是没数据，就优先检查：

- `rx_id` 是否配置正确
- `can_handle` 是否对应正确控制器
- 回调实例是不是注册到了预期对象

### 8.3 如果“明明发出去了却报错”，先看 TX 中断策略

重点检查：

- `CAN_SetInterruptEnable()` 是否传入了 `CAN_INTERRUPT_ALL`
- `INT_MASK` 是否放行了 `TX_FINISH` / `TX_FAIL`
- `CAN_Common_IRQHandler()` 是否有机会更新 `s_canTxDone/s_canTxFail`

---

## 9. 本次最终有效配置

本轮联调最终确认有效的测试配置为：

- MCU TX ID：`0x200`
- MCU RX ID：`0x201`
- 驱动默认中断：`CAN_INTERRUPT_ALL`
- 应用初始化顺序：

```c
CAN_SetInterruptEnable(CAN_INTERRUPT_ALL);
CAN_Service_Init();
App_CanInit();
```

在这个配置下，当前版本已经可以稳定完成本轮 CAN 联调。
